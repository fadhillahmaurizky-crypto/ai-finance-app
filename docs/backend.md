# Backend

No traditional application server. "Backend" means four things: Supabase (Postgres + custom RPC auth), four Vercel serverless functions (AI proxy + Xendit token top-up), and a Google Apps Script Fonnte WhatsApp webhook (see §3/§4).

## 1. Supabase — primary backend

The app talks to PostgREST via a small wrapper in `ui-helpers.js`:

```js
function authToken(){ return localStorage.getItem('sdk_token') || SB_KEY; }
async function sb(path, method='GET', body=null){ /* Authorization: Bearer authToken() */ }
async function rpc(fnName, params={}){ /* POST /rest/v1/rpc/fnName, same auth */ }
```

After login, `authToken()` returns the custom-signed JWT (not the bare anon key) — this is what RLS policies actually check. Before login (or for a logged-out visitor), it falls back to the anon key, which is fine since the only things reachable pre-login are `login_check`, `create_password_reset`, `confirm_password_reset`, and self-registration (all `SECURITY DEFINER` or explicitly anon-scoped).

**Auth is not Supabase Auth.** See `database.md`'s "Custom auth & RLS" section for the full design and the reasoning behind it (short version: it was cheaper to hand-roll a JWT than migrate the whole user model, and it reuses the existing `users` table, EmailJS OTP flow, and admin tooling unchanged).

`sb()`/`rpc()` both handle `401` by calling `handleAuthExpired()` (in `app-core.js`), which clears the session and redirects to login with a toast — this fires if the JWT is invalid/expired (30-day expiry, no refresh flow yet — see `roadmap.md`).

## 2. Vercel serverless functions (`/api`) — this is where the Groq key lives now

`/api/ai-chat.js` and `/api/ai-scan.js` are the **only** things that ever see `GROQ_API_KEY` (a Vercel environment variable, never in any file in this repo). Both:
1. Take a `user_id` from the request body.
2. Look up that user's `plan`/`role`/`tokens_used`/`tokens_limit` **server-side** via Supabase, using `SUPABASE_SERVICE_ROLE_KEY` (so plan/limit gating can't be bypassed by a modified client).
3. Call Groq with the server-side key.
4. Return just the result (`reply`/`tokensUsed` for chat, `content`/`tokensUsed` for scan).

**Why service role, specifically, for step 2**: after the RLS migration (`database.md` block `[18]`), every table's policy requires `is_owner_or_admin(user_id)` — a claim proven by a real signed JWT. These two functions never receive a user's JWT, only a plain `user_id` string in the request body, so an anon-key-authenticated lookup here satisfies no RLS policy and silently returns zero rows — **this broke both AI features completely** (`"User tidak ditemukan"` for every user, every request) from the moment that RLS migration shipped until this was caught and fixed. `SUPABASE_SERVICE_ROLE_KEY` bypasses RLS by design, which is exactly appropriate here: the Vercel function itself is the trusted party doing a legitimate server-side authorization check, not relaying a user's own request. Its use is deliberately scoped to just this one lookup in each file — not a general-purpose shortcut, and it must never reach client-facing code.

Same-origin (both the static site and `/api` are on the same Vercel deployment), so there's no CORS complexity — this was a deliberate choice over routing AI calls through Google Apps Script, which has real, well-documented CORS/response-readability problems for this kind of browser-facing call.

**Before this existed**, the Groq key was fetched by the client from a `settings` table row using the public anon key — which meant *anyone* who opened devtools on the live site could read it directly from the database, no login required. That table is now fully locked down (`FOR ALL USING (false)`) and nothing reads it anymore.

## 3. Google Apps Script (`gs/fonnte.gs`) — real, committed, Fonnte WhatsApp webhook

**`gs/fonnte.gs` exists in this repo.** Earlier drafts of these docs said no `.gs` file existed anywhere in the repo (not even in git history) — that was true once, but is no longer the case; don't rely on that older claim. Deployment to the live Apps Script editor is a manual copy-paste step the product owner does themselves — committing the file here keeps it in version control and reviewable, but doesn't itself push anything live. **Not necessarily running the exact version committed here** at any given moment, for that reason.

**What it does**: inbound WhatsApp message (via Fonnte's webhook, `doPost`) → look up the user by `wa_number` (`getUserIdByWA()`) → fetch that user's real `accounts` and `user_categories` once per message → for each line: extract the amount (`extractNominal()` — understands `rb`/`ribu`/`jt`/`juta`/`k` suffixes), match an account name mentioned anywhere in the line as a whole word (`matchFromText()`, falls back to the user's default account if none mentioned), classify `jenis`/`kategori` against the user's *real* categories first, a keyword heuristic second (`analisaDenganAI()`), build a clean `keterangan` with the matched amount/account text stripped out (`buildKeterangan()`) → insert the transaction (now correctly attributed to `account_id`) → reply with the updated all-time Saldo.

**Fixed this session** (previously real, tracked bugs — see git history / the fix's PR for the full before/after):
- **Auth**: was using the public anon key for every Supabase call (`SB_KEY`, hardcoded) — since the RLS migration, that satisfies no table's `is_owner_or_admin(user_id)` policy for any row, so `getUserIdByWA()`/`saveTrxToSupabase()` almost certainly failed silently for every real user. Now uses a Supabase **service role** key (bypasses RLS by design, same reasoning as `/api/ai-chat.js`/`/api/ai-scan.js`), stored as a Script Property (`SUPABASE_SERVICE_ROLE_KEY`) — never hardcoded, matching how `FONNTE_TOKEN` was already handled.
- **`account_id`**: transactions logged via WhatsApp never set it, breaking per-account balance tracking. Now resolved per-line (matched account name in the message, or the user's default account) and always set.
- **Balance model**: `getSaldoSupabase()`/`getLaporanSupabase()` computed "saldo" as the current month's transactions only, while the app itself uses an all-time model (`Σ(account.saldo_awal)` + all-time pemasukan − all-time pengeluaran, transfers excluded) — a user could get a different "saldo" answer from the bot than from the app for the same account. Both now compute the same all-time formula; `getLaporanSupabase()` keeps a month-scoped income/expense breakdown as a clearly-separate section, not labeled "Saldo."
- **Category vocabulary**: `analisaDenganAI()` used the *original* category set from early in the project (`makanan`/`transportasi`/`hiburan`/`tagihan`) instead of the app's current defaults (`makan`/`belanja`/`elektronik`/`pulsa`/`paket_data`) — WhatsApp-logged transactions created duplicate, differently-named categories from the app's own. Now matches the user's actual `user_categories` (default *and* custom) directly first, falling back to an updated keyword heuristic mapped to real default category names only if no direct match — not a hardcoded list disconnected from what the app actually seeds.
- **Account-name extraction** (addendum spec): a message like `"makan bubur 10rb BCA"` now resolves to `keterangan:"makan bubur"`, `account: BCA` (matched by name, whole-word, case-insensitive, longest match wins if ambiguous) instead of storing the whole raw line as `keterangan` and always defaulting to the user's default account.
- Left as **not urgent**, per the spec's own call: OTP delivery is still GET-based (`doGet` → `sendOTPLogin`) — no client code currently calls this action at all (registration/forgot-password OTP both go through EmailJS instead), so this is genuinely dead code right now, not worth changing until something actually calls it.

The client (`js/config.js`'s `GAS` constant, a real deployed Apps Script Web App URL — **not necessarily running `gs/fonnte.gs`'s exact current content**, since deploying to Apps Script is a manual copy-paste step, not tied to this repo's git history) also calls it for:
- **`?action=ping`** — liveness check, `pingSheetSync()` in `settings.js` (autosync heartbeat), fire-and-forget (`mode:'no-cors'`, the client can't read the response).
- **`?action=notifyAdmin&msg=...`** — called from `payment.js`'s `submitPayment()`, which is itself orphaned as of the trial-registration change (see `features.md`'s "Payment / Plan-Selection Flow" entry) — not reachable from any current UI.

**`admin.html` used to also call `?action=adminData`** (a supposed Google-Sheets-backed platform-wide transaction summary for the Laporan tab) — this action was never actually implemented anywhere, live-deployed or committed, so it always returned `{"error":"Action tidak dikenal"}` and the Laporan tab was permanently blank, confirmed by testing the live endpoint directly while real transaction data existed in Supabase the whole time. Fixed by having `renderLaporan()` query `transactions`/`accounts` directly instead — Supabase is already the authoritative datastore for this data, so routing it through Apps Script/Sheets was an unnecessary extra hop that quietly broke. **If a future feature is tempted to add a new GAS action that mirrors data Supabase already owns, query Supabase directly instead** — this is the second time in this project a GAS/Sheets-mediated copy of live Supabase data has turned out to be the actual bug (see the `gas.saldo`/`gas.totalTrx` fields still used elsewhere in `admin.html`'s Manajemen User table and Detail modal, which have the same latent risk if that Sheets sync isn't actually running).

## 4. Xendit (`/api/create-payment.js` + `/api/xendit-webhook.js`) — token top-up, test mode

Per `wangku-spec-xendit-topup-testmode.md`. Scoped specifically to **buying extra AI tokens**, a self-service, frequent-small-purchase flow that manual-transfer-plus-admin-approval doesn't scale to. Subscription plan purchases (Basic/Pro) are explicitly **not** in scope — they keep using the existing `orders` table + manual approval flow (`payment.js`) unchanged, which is why token top-ups get their own separate `token_purchases` table (block `[28]`) instead of extending `orders`.

**Two-step flow, same "secret never reaches the client" pattern as Groq:**
1. **`POST /api/create-payment`** — client sends `{user_id, package}` (`'2jt'` or `'5jt'`, matching the tiers already live in `admin.html`'s manual token-grant UI: 2 juta token/Rp 35.000, 5 juta token/Rp 59.000). The server looks up the actual tier server-side (hardcoded, never trusts a client-supplied amount — same reasoning as the plan/token gating in `ai-chat.js`/`ai-scan.js`), inserts a `'pending'` `token_purchases` row, then calls Xendit's Invoice API (`POST https://api.xendit.co/v2/invoices`, HTTP Basic Auth with `XENDIT_SECRET_KEY_TEST` as the username) using that row's own `id` as `external_id`. Returns `{checkout_url}` — a Xendit-hosted payment page the client redirects to. No Xendit SDK or key ever reaches the browser.
2. **`POST /api/xendit-webhook`** — Xendit calls this once the invoice is actually paid. This is the **source of truth**, not the client returning from checkout (that redirect can be skipped, delayed, or the URL guessed at by anyone — it proves nothing about whether payment actually happened). Two things happen in order:
   - **Signature check first, before anything else is trusted**: compares the `x-callback-token` request header against `XENDIT_WEBHOOK_VERIFICATION_TOKEN_TEST` (a separate credential from the API secret key, found in Xendit's dashboard under Settings → Developers → Webhooks). A mismatched or missing token gets a `401` immediately — the payload is never even parsed for business logic past that point.
   - **Idempotent grant**: `UPDATE token_purchases SET status='paid' WHERE id=<external_id> AND status='pending'`. Xendit can and does redeliver the same callback (retries) — this `WHERE status='pending'` clause *is* the idempotency guarantee, no separate locking needed. A redelivered callback matches zero rows (the row's already `'paid'` from the first delivery) and the function just acknowledges `200` without granting anything a second time. Only when a row is actually flipped does the function grant tokens — **additively**: fetches the user's current `tokens_limit`, then `PATCH`es `tokens_limit = current + purchased` (`tokens_used` untouched). **Deliberately not the same as `admin.html`'s `setUserTokens()`**, which replaces `tokens_limit` outright and resets `tokens_used` to `0` — that's a manual override/correction action with different intent than a self-service top-up, and the "Beli Token **Tambahan**" ("buy **additional** tokens") label in Settings explicitly promises addition, not replacement. The first version of this function got this wrong (matched `setUserTokens()`'s replace semantics instead) — caught via a real test-mode purchase where existing tokens visibly got wiped instead of added to, not by code review. Fixed by reading the current limit before writing the new one, since PATCH can't express "add to the existing value" directly.

**Client side** (`js/settings.js`): a "Beli Token Tambahan" mini-section in the Pro-plan card (`renderPlanCard()` in `app-core.js`) replaces what used to be a single "Isi Token Lagi" button that opened WhatsApp with a stale, wrong price (`Rp 29.000`, matching neither real tier) — now two real tier buttons calling `buyTokenPackage()`. After returning from the Xendit-hosted checkout (`?xendit_return=success|failed` on the redirect URL), `checkXenditReturn()` — called once per `showApp()` — shows a "Memproses pembayaran..." toast and polls `get_user_by_id` every 3 seconds (up to 6 times) rather than assuming `tokens_limit` is already updated the instant the user is back, since the webhook can land a few seconds after the redirect.

**Verified two ways.** Directly against the database, without live credentials: inserted a real `'pending'` row, ran the exact `UPDATE ... WHERE status='pending'` the webhook runs — first attempt returns the row and flips it to `'paid'`, an identical second attempt (simulating a Xendit retry) returns zero rows, confirming a token grant can't double-fire; confirmed `token_purchases` is genuinely unreadable via a plain anon-key REST call (`200` + empty array, same as `password_reset_tokens`); and confirmed the additive-grant fix specifically with a partial-usage scenario (`tokens_limit=2,000,000`/`tokens_used=1,500,000`, bought `+5jt` → correctly became `tokens_limit=7,000,000`/`tokens_used=1,500,000` unchanged, i.e. remaining went from 500,000 to 5,500,000). Separately, **the product owner completed a real end-to-end test-mode purchase** through the actual UI (Settings → tier button → real Xendit-hosted checkout → real sandbox payment → webhook fired → tokens updated) — this is what surfaced the replace-vs-additive bug in the first place, and confirmed everything else in the flow (invoice creation, checkout redirect, webhook delivery, signature verification) genuinely works, not just in theory.

### 4a. Payment history & cancellation handling — addendum

Per `wangku-spec-xendit-payment-history.md`, a genuine gap in the original spec above: nothing showed a user their purchase status/history, including purchases that were abandoned or failed rather than completed. Three additions, no new table:

- **`EXPIRED` handling in the webhook**: Xendit auto-expires an unpaid invoice after its configured window and fires this same webhook with `status: 'EXPIRED'` instead of `'PAID'` — previously silently ignored (`ignored: 'EXPIRED'`, ack'd with `200`, nothing else happened), which is exactly why an abandoned checkout stayed `'pending'` forever with no visibility. Now handled with the identical idempotent-UPDATE shape as the `PAID` path (`UPDATE ... SET status='expired' WHERE id=<external_id> AND status='pending'`), just without a token grant. Verified a redelivered `EXPIRED` callback is a no-op (matches zero rows, same as `PAID`'s retry case), and separately that a late/stray `PAID` callback arriving *after* a row has already expired is also correctly a no-op — both terminal states share the same `WHERE status='pending'` guard, so whichever webhook call lands first "wins" and the other can never override it.
- **`failed` set at creation time, not just via webhook**: if the Xendit invoice-creation API call in `/api/create-payment.js` itself errors (`!invoiceRes.ok`), the `'pending'` row already inserted moments earlier is immediately marked `'failed'`. This is a different failure mode than expiry — no invoice was ever actually created, so no webhook of any kind is ever coming for that row. Without this, it would've been indistinguishable from a genuinely in-flight purchase in the history UI.
- **Read access for the row's own owner**: `token_purchases` was fully unreadable by design (see §4 above) — the new "Riwayat Pembayaran" page in Settings needed a way to show a user their own rows without loosening that lockdown for anyone else's. Rather than an RPC, this uses a second, narrower RLS policy (block `[30]`, see `database.md`) scoped to `FOR SELECT` only, added *alongside* the original `FOR ALL USING (false)` policy — Postgres OR's permissive policies together per-command, so `SELECT` becomes `is_owner_or_admin(user_id)` while every other command still only sees `false`. `showPaymentHistory()` (`js/settings.js`) then fetches directly (`sb('token_purchases?user_id=eq....&order=created_at.desc')`), same pattern as every other user-owned table — no new RPC needed. Package labels (`tokenPurchaseLabel()`) are derived client-side from the `tokens` count rather than stored as a separate column, since there are only two tiers and the mapping is already hardcoded once in `create-payment.js`.

**Verified**: created one `token_purchases` row per status (`pending`/`paid`/`expired`/`failed`) for a temp test user plus one `paid` row for a second temp user, logged in as the first and confirmed the history modal renders exactly the 4 owned rows (correct label/amount/date/status badge per row, newest first) and **not** the second user's row — confirming the RLS SELECT policy is scoped correctly, not just present. Also confirmed the empty state ("Belum ada riwayat pembelian") for a user with zero purchases. Both temp users and all rows deleted after.

### 4b. Subscription renewal (Basic/Pro) — manual-but-digitized

Per `wangku-spec-subscription-renewal.md`. Closes a different gap from 4a: once a user was set to `plan='basic'`/`'pro'` (via the existing manual transfer-proof + admin approval flow), nothing tracked when that period ended — only the *trial* had a real expiry (`trial_ends_at`). `plan_expires_at` (block `[31]`, `database.md`) extends the exact same lazy-check pattern to paid periods. Explicitly **not** built: any grace period after expiry, or true auto-recurring/stored-payment-method billing — each renewal is a fresh, deliberate payment, same principle as the trial's own all-or-nothing lazy downgrade.

**Two payment paths, one shared formula, both converging on the same result** (deliberately not a third table or a new RPC — see `database.md`'s "Subscription renewal mechanics" for the exact rule):
1. **Xendit** (self-service, new): `/api/create-payment.js` now accepts `item_type:'plan'` alongside the existing token-package purchases (`item_type:'tokens'`, the default) — same endpoint, same `token_purchases` table, distinguished by that one parameter rather than a parallel system. `PLAN_PACKAGES` (`basic`: Rp 19.000, `pro`: Rp 39.000, both bundling that plan's monthly token allotment) mirrors `PLANS` in `config.js` and `konfirmasiOrder()`'s tier→tokens mapping in `admin.html` — one more place that has to be kept in sync if pricing ever changes, same caveat as the token tiers in 4/4a. On webhook confirmation, the grant branches on `purchase.item_type`: `'plan'` rows extend-or-set `plan_expires_at`, reset `tokens_limit`/`tokens_used`/`token_reset_month` fresh for the new period (**not additive** — a renewal starts a new billing cycle, unlike the token top-up in §4, which is deliberately additive mid-cycle), and unconditionally clear `trial_ends_at`.
2. **Manual approval** (existing, extended): `admin.html`'s `konfirmasiOrder()` — the transfer-proof + admin-approval flow — now computes and sets `plan_expires_at` with the identical extend-or-set formula, and clears `trial_ends_at` the same way. Kept working as a deliberate alternative to Xendit, not replaced — some users may not be able to pay via Xendit's supported methods.

**Client-side**: `buyPlanRenewal(planId)` (`js/settings.js`) mirrors `buyTokenPackage()` exactly — POSTs to `/api/create-payment` with `item_type:'plan'`, redirects to the returned Xendit checkout URL. Wired from two places: a "Perpanjang Sekarang" button on the Basic/Pro plan card (shown whenever `plan_expires_at` is set — i.e., a real paid period, not an admin's permanent "Ubah Plan" override) and a renewal nudge on Home reusing the trial nudge's exact 0-2-day-window pattern (`renderTrialNudge()` in `dashboard.js`, falls through to the renewal case once not on trial). `checkXenditReturn()`'s success detection had to change too — it used to only check `tokens_limit` increased, which doesn't reliably signal success for a plan purchase (Basic bundles 0 tokens; a Pro renewal can reset `tokens_limit` to the *same* number it already was) — now also treats `plan_expires_at` moving later as a success signal.

**Verified**: five scenarios against temp test users (deleted after) rather than a live purchase (test-mode Xendit checkout itself was already verified end-to-end in §4; this session's verification focused on the renewal-specific arithmetic and lazy-downgrade, simulating the webhook's exact logic via direct SQL where a live purchase wasn't necessary to prove it):
- Fresh plan purchase (no prior `plan_expires_at`): correctly set exactly 30 days out.
- Early renewal (5 days still remaining on the current period): correctly became 35 days out (5 remaining + 30 new) — proving it extends from the *existing* expiry, not from today.
- An already-expired `plan_expires_at`: a real login through `login_check` (not a simulation) correctly downgraded `plan` to `'free'` in the database itself (not just the RPC's returned object), leaving `tokens_limit`/`tokens_used` untouched.
- A user mid-trial (`trial_ends_at` in the future) paying for a real plan: correctly cleared `trial_ends_at` to `NULL` while setting a fresh `plan_expires_at`.
- The admin Detail User modal, via a real admin login: correctly rendered "Pro — berakhir 20 Jul 2026, 5 hari lagi" for a paid non-trial subscriber (`planExpiryInfo()`, mirroring `trialInfo()`).

Also hit a **local tooling quirk worth remembering**, not a code bug: this app's service worker (`sw.js`) is network-first by design (`fetch(e.request).then(...).catch(() => caches.match(...))`), but the preview browser's own script-tag caching still served a stale `app-core.js` across several reloads after edits, even past a full service-worker unregister + cache clear. Worked around by fetching the file with a cache-busting query and `eval`-ing it directly to confirm the logic, rather than chasing the exact caching layer responsible — real deployments get a normal fresh load per Vercel's own cache-control headers, this was specific to the local static preview server.

**Test-mode → live-mode switch (not built, deliberately designed for)**: `XENDIT_SECRET_KEY_TEST`/`XENDIT_WEBHOOK_VERIFICATION_TOKEN_TEST` are named with `_TEST` specifically so that adding `_LIVE` counterparts later and pointing the two functions at them is a one-line env var change, not a code change — see `environment.md`.

## What's NOT server-side
- No image resizing/compression before upload (receipt photos, avatars) — sent as full-size base64.
- No rate limiting beyond the plan/token check inside the two Vercel functions.

## What IS server-side (beyond auth/RLS)
- **Per-account balance enforcement** — a Postgres trigger (`wangku_check_account_balance()`, block `[21]`) blocks a `pengeluaran`/`transfer` from pushing its source account negative, as a safety net behind the client-side check in `js/transactions.js`/`js/accounts.js`. See `database.md`.
